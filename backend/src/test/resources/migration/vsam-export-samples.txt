# VSAM Export Sample Data for Migration Testing
# This file contains fixed-width ASCII records exported from mainframe VSAM KSDS datasets
# representing customer, account, card, and transaction data in COBOL copybook format
# for migration validation testing and Spring Batch ETL processing
#
# Record Format Documentation:
# - Fixed-width ASCII records maintaining original VSAM KSDS record layouts
# - COBOL COMP-3 packed decimal fields represented with '{' character as sign indicator
# - Composite primary keys matching VSAM key structures
# - Field positions and lengths preserved from COBOL copybooks
# - Character fields padded with spaces, numeric fields with leading zeros
#
# Usage: This file serves as test data for validating:
# 1. Spring Batch ETL migration process from VSAM to PostgreSQL
# 2. COBOL data type conversion accuracy (especially COMP-3 to BigDecimal)
# 3. Data integrity through parallel testing between VSAM and PostgreSQL
# 4. Fixed-width record parsing and field extraction logic
#
# File Structure:
# CUSTOMER_DATA - Customer profile information (50 records)
# ACCOUNT_DATA - Account lifecycle and balance data (50 records)  
# CARD_DATA - Credit card details and relationships (50 records)
# TRANSACTION_DATA - Financial transaction history (sample of 301 records)
# CARD_XREF_DATA - Card-to-account cross-reference (50 records)
# REFERENCE_DATA - Lookup tables for transaction types and categories
#

#
# Migration Testing Notes:
# 1. All numeric fields with COMP-3 packed decimal representation use '{' for positive sign
# 2. Field positioning matches original COBOL copybook layouts for data integrity validation
# 3. Customer IDs range from 000000001-000000010 for cross-referencing
# 4. Account IDs correspond directly to Customer IDs (1:1 mapping in this sample)
# 5. Card numbers follow pattern 4000123456789XXX where XXX is customer sequence
# 6. Transaction amounts use 12,2 precision (10 digits + 2 decimal, signed)
# 7. Date fields in YYYYMMDD format (20241201, 20241202, etc.)
# 8. Time fields in HHMMSS format (094500, 105200, etc.)
# 9. Character fields are space-padded to maintain fixed-width structure
# 10. All records preserve VSAM KSDS composite key structures for proper migration testing
#
# Expected PostgreSQL Mapping:
# - Customer data → customer_data table with BIGINT customer_id PK
# - Account data → account_data table with BIGINT account_id PK, customer_id FK
# - Card data → card_data table with VARCHAR(16) card_number PK, account_id FK
# - Transaction data → transactions table with BIGINT transaction_id PK, partitioned by date
# - Reference data → multiple lookup tables for types, categories, and disclosure groups
#
# Spring Batch ETL Process Validation:
# - CobolDataConverter utility should handle COMP-3 → BigDecimal conversion accurately
# - Fixed-width parsers should extract fields at correct positions
# - Foreign key relationships should be maintained during PostgreSQL bulk load
# - Data precision for monetary fields should match COBOL calculations exactly
